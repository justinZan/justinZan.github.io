# 全栈面试：项目经验

## 如何介绍一个全栈项目？

推荐结构：

1. 项目背景
2. 你的职责
3. 技术栈
4. 核心功能
5. 难点和解决方案
6. 结果和收益
7. 复盘和改进

不要只说“我做了页面和接口”。更好的表达是：我负责从需求拆解、前端交互、接口设计、数据表设计到部署上线的完整链路。

## 如何讲项目难点？

一个好的项目难点要包含：

- 场景
- 问题
- 方案
- 取舍
- 结果

示例：

```text
当时列表数据量比较大，页面滚动卡顿。
我先用性能工具定位到 DOM 节点过多。
后来改成虚拟列表，并把筛选接口做了分页。
最终首屏渲染时间明显下降。
```

## 如何回答“你在项目中负责什么”？

可以这样组织：

- 我负责前端页面和组件开发。
- 我参与接口设计和联调。
- 我设计了部分数据库表结构。
- 我处理了登录态和权限校验。
- 我参与部署和线上问题排查。

如果是全栈岗位，要突出端到端能力，而不是只讲单个页面。

## 如何回答“遇到过什么线上问题”？

可以按事故复盘思路回答：

1. 问题现象
2. 影响范围
3. 临时止血
4. 根因定位
5. 最终修复
6. 后续预防

示例方向：

- 接口超时
- 缓存击穿
- 前端白屏
- 登录态失效
- 数据库慢查询
- 发布配置错误

## 如何回答“如何保证代码质量”？

常见措施：

- TypeScript
- ESLint / Prettier
- 单元测试
- 接口测试
- Code Review
- 组件抽象
- 错误边界
- 日志和监控
- 灰度发布

回答时最好结合项目：哪些地方用了自动化，哪些地方靠 review，哪些地方有线上监控。

## 如何回答“如何和产品、设计、后端协作”？

可以从流程讲：

- 需求阶段确认边界和验收标准。
- 设计阶段确认交互和异常状态。
- 开发阶段同步接口字段和错误码。
- 联调阶段对齐 mock、环境和测试数据。
- 上线前确认回归范围。
- 上线后关注日志和用户反馈。

全栈工程师的优势是能理解各端约束，减少沟通断层。

## 如何回答“你的技术优势是什么”？

可以从全栈角度回答：

- 能从用户体验理解需求。
- 能设计前后端接口和数据结构。
- 能独立定位端到端问题。
- 能关注性能、安全和可维护性。
- 能把复杂功能拆成可交付的小任务。

不要只说“学习能力强”，最好给一个具体例子。

## 常见行为面试题

### 介绍一个你最有成就感的项目。

可以按“背景、目标、职责、难点、方案、结果”来讲。不要只说项目用了什么技术，而要说明你解决了什么业务问题。

例如：我负责一个后台管理系统的核心模块，原来列表查询慢、表单维护困难。我重构了查询接口和前端表单配置，增加分页、缓存和错误提示，最终页面响应明显变快，后续新增字段也更容易。

### 介绍一次你解决复杂问题的经历。

复杂问题要体现排查思路，而不是只说最后修好了。可以先描述现象，再说明你如何分层定位：前端、接口、数据库、缓存、网络、第三方依赖。

成熟回答会包含：用了哪些日志和监控、如何缩小范围、如何临时止血、最终根因是什么、后续如何防止再次发生。

### 你如何处理需求变化？

先确认变化的原因和目标，再评估影响范围，包括页面、接口、数据、测试和上线时间。然后把方案拆成必须做、可以延期、需要确认的部分。

如果变化影响交付，要及时同步风险，而不是默默加班硬扛。面试时可以强调：我会尽量支持业务变化，但会把成本和风险透明化。

### 你如何处理和同事意见不一致？

先把分歧从“谁对谁错”转成“目标和约束是什么”。比如性能、开发效率、可维护性、安全、上线时间，哪个更重要。

如果仍然难以判断，可以用小实验、数据或技术评审来决策。回答时要体现协作态度：尊重对方经验，但最终以项目目标和事实为准。

### 你如何学习新技术？

我通常先看官方文档和核心概念，再做一个最小 Demo，最后结合项目场景判断它能解决什么问题。学习不是为了堆名词，而是为了提升交付质量。

如果要引入团队项目，还会评估生态、维护成本、迁移成本、团队熟悉度和风险，并沉淀成文档或示例。

### 你如何安排任务优先级？

优先级可以按业务价值、紧急程度、依赖关系和风险来排。会先保证核心链路、阻塞他人的任务和上线风险高的任务。

具体执行时我会把大需求拆成小任务，先交付可验证的版本，再迭代细节。遇到时间不足，会主动同步取舍方案。

### 你如何面对线上故障？

线上故障先止血，再定位根因。止血方式包括回滚、降级、限流、切换配置、扩容或临时关闭问题功能。

恢复后再分析日志、监控和变更记录，找到根因，补测试、监控、告警和复盘。面试时要避免说“先慢慢查原因”，核心是先降低用户影响。

### 你希望加入什么样的团队？

可以回答得真诚但职业化：希望团队目标清晰、沟通直接、重视工程质量，也愿意给成员成长和承担责任的机会。

同时可以补充自己的贡献：我希望在这样的团队里不仅完成需求，也能主动改善性能、稳定性、文档和协作流程。

## 项目回答英文模板

```text
In this project, I was responsible for both the frontend and part of the backend.
I built the main user interface, designed several API endpoints, and worked on the database schema.
One challenge was performance. The page became slow when the data size grew.
I solved it by adding pagination, optimizing the query, and reducing unnecessary rendering.
This experience helped me understand the whole process from user experience to backend performance.
```
